# Phase 5: Factory Cluster (Core Product) - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

The factory cluster accepts a natural-language goal and produces a fully runnable, self-contained cluster artifact: YAML agent configs, docker-compose.yml, generated Dockerfile, requirements.txt, seeded database, and launch.sh. The factory cluster IS a cluster — it uses the same boss/worker/heartbeat/state-machine machinery to produce its output. Creating posts, running product clusters, or implementing tool execution hooks are separate phases.

</domain>

<decisions>
## Implementation Decisions

### Factory Architecture (Core)
- The factory cluster is self-similar: it uses BossAgent + WorkerAgent + BaseAgent heartbeat machinery to generate cluster artifacts, the same as any product cluster
- Factory workers are specialized for artifact generation: **researcher**, **security-checker**, **executor** (plus boss and critic as structural constants)
- `agent-factory create` is fire-and-forget: seeds the factory DB with the goal, starts factory heartbeats, returns immediately
- User polls progress with `agent-factory status <name>`

### Role Decomposition Pipeline
Multi-step LLM pipeline run by the factory boss (decomposes into tasks for factory workers):

1. **Responsibilities generation** — LLM enumerates all responsibilities the goal requires
2. **Clustering** — responsibilities grouped into roles; strongly overlapping ones combined to avoid gaps
3. **Fit check** — LLM rubric evaluates each role for: responsibility cohesion, privilege scope, name clarity. Max 2 retries with failure reason fed back into re-clustering
4. **Enrichment** (parallelizable per role) — LLM assigns: personality tone, descriptive name, tool_allowlist from researcher-determined catalog
5. **Structured output** — `RoleSpec` Pydantic model via `messages.parse()` (same pattern as BossAgent's goal decomposition). Fields: `name`, `responsibilities`, `personality_system_prompt`, `tool_allowlist`, `requires_glibc`
6. **Deterministic YAML generation** — executor writes `agents/<role>.yaml` from validated `RoleSpec` (no free-form YAML from LLM)

### Structural Role Constants
- **Boss**: always generated, not subject to decomposition pipeline
- **Critic**: always generated with fixed skeptical/adversarial personality + goal-specific context injected into system prompt. Purpose: peer review quality depends on a dedicated skeptic

### Personality
- Woven into system prompt, generated by LLM (not a separate field)
- Intentional and role-appropriate: e.g., "cynical/snide critic who pokes holes in every deliverable", "curious/energetic researcher who investigates thoroughly"
- Factory generates distinct personalities per role so agents have genuine character

### Tool Allowlist and Dependencies
- Factory's **researcher agent** investigates what tools/libraries the goal requires (e.g., PyMuPDF for PDF extraction, requests for web fetch)
- Factory's **security-checker agent** audits each tool for: privilege scope, glibc dependency, known CVEs
- `requirements.txt` generated in cluster artifact with all researcher-determined packages
- Tool allowlist in generated agent YAMLs is declarative for Phase 5 — execution hooks (Claude tools API wiring) are future work
- If security-checker flags a tool as requiring glibc: Dockerfile switches from slim to Ubuntu base

### Gap Detection (Post-Creation)
- The product cluster's boss detects role gaps during its own heartbeat and logs to `activity_log`
- Gap-filling is user-initiated: `agent-factory add-role <cluster> <role-description>` (Phase 5 adds this CLI command)
- The command runs a mini factory pipeline: generates a new `agents/<role>.yaml`, updates `docker-compose.yml`, notifies user to restart the service

### Cluster Naming
- `--name` flag for explicit naming; auto-slugified from goal string as fallback (e.g., "Analyze PDFs" → `analyze-pdfs`)
- Collision policy: fail fast with clear error message unless `--force` flag is passed (overwrites existing directory)
- Factory home: `FACTORY_HOME` env var, default `~/.agent-factory/`; contains factory DB and factory agent configs
- Cluster output: `./clusters/<name>/` relative to the working directory where `agent-factory create` is run

### Dockerfile / Image Strategy
- Each generated cluster includes a `Dockerfile` — truly standalone, no dependency on the factory repo
- Base image: `python:3.12-slim` by default
- If security-checker flags a tool dependency requiring glibc (e.g., compiled C extensions): Dockerfile switches to `ubuntu:22.04` with `python3.12` installed
- `requirements.txt` is `COPY`'d into the Dockerfile and `pip install`'d at `docker compose build` time — not at container startup
- `launch.sh` runs `docker compose build` then `docker compose up -d` with env var validation

### CLI Feedback

**`agent-factory create <goal> [--name <name>] [--force]`** (fire-and-forget):
```

  Factory DB: ~/.agent-factory/factory.db
  Track progress: agent-factory status analyze-pdfs
  Cluster output: ./clusters/analyze-pdfs/ (when complete)
```

### Claude's Discretion
- Exact task breakdown the factory boss uses to decompose a cluster-creation goal (task titles, count, assignment strategy)
- Internal structure of `factory/generator.py` utility functions
- How the executor worker materializes documents as files on disk (tool call vs direct file write)
- Fit check rubric scoring thresholds (what constitutes a failing score)
- Auto-slug truncation behavior for very long goal strings

## Existing Code Insights

### Reusable Assets
- `runtime/heartbeat.py` — `BaseAgent`: factory cluster boss and workers subclass this directly; no changes needed
- `runtime/boss.py` — `BossAgent`: factory boss IS a BossAgent; reuse goal decomposition, peer review promotion, stuck detection, gap-fill cron
- `runtime/worker.py` — `WorkerAgent`: factory researcher, security-checker, executor all subclass this; role-specific system prompts configure behavior
- `runtime/config.py` — `AgentConfig` + `load_agent_config()`: factory agents use the same YAML config pattern; `FACTORY_HOME/config/` holds factory agent configs
- `runtime/cli.py` — `factory_cli` stub: already registered as the `agent-factory` entry point in `pyproject.toml`; Phase 5 adds `create`, `list`, `status`, `add-role` subcommands
- `runtime/schema.sql` — canonical schema: copied verbatim into each generated `clusters/<name>/db/schema.sql`
- `runtime/models.py` — `_uuid()`, `_now_iso()`: reused in factory DB seeding
- Phase 3 CLI patterns: `--db-path` / `CLUSTER_DB_PATH` env var, tabulate table output, `--json` flag, `asyncio.run()` once per command

### Established Patterns
- `messages.parse()` for structured LLM output (BossAgent's `decompose_goal` uses this): `RoleSpec` model follows same pattern
- Base + overlay YAML merge (`{**cluster_raw, **role_raw}`): factory's own agent configs use this; generated clusters inherit it
- `pytest.importorskip` inside test body: factory/ module is absent until Wave 1; same stub pattern for test collection safety
- Tabulate `"simple"` tablefmt: consistent across all CLI commands

### Integration Points
- `runtime/cli.py:factory_cli` — Phase 5 fills in `create`, `list`, `status`, `add-role` subcommands
- `pyproject.toml` — `factory/` package needs to be added to `[tool.hatch.build.targets.wheel] packages`
- Factory's own DB lives at `FACTORY_HOME/factory.db` (separate from any cluster DB)
- Generated `clusters/<name>/runtime/` is a copy of the current `runtime/` package directory

</code_context>

<specifics>
## Specific Ideas

- The factory cluster is self-similar — it's a running demonstration that the runtime works; Phase 5 is implicitly Phase 6's integration test
- Personality should be intentional and character-driven: the critic is cynical/snide, the researcher is curious/energetic. This is a product differentiator, not a cosmetic detail
- The multi-step role decomposition pipeline (responsibilities → cluster → fit check → enrich → structured output → YAML) was explicitly preferred over free-form YAML generation from LLM — keep this precise even when it adds LLM call count
- `launch.sh` must validate `ANTHROPIC_API_KEY` is set before calling `docker compose up` — hard requirement from REQUIREMENTS §5
- Factory status command reuses the same tabulate table format as `cluster tasks list` — visual consistency

</specifics>
