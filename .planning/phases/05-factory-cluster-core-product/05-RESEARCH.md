## User Constraints (from CONTEXT.md)

### Locked Decisions

**Factory Architecture (Core)**
- The factory cluster is self-similar: it uses BossAgent + WorkerAgent + BaseAgent heartbeat machinery to generate cluster artifacts, the same as any product cluster
- Factory workers are specialized for artifact generation: **researcher**, **security-checker**, **executor** (plus boss and critic as structural constants)
- `agent-factory create` is fire-and-forget: seeds the factory DB with the goal, starts factory heartbeats, returns immediately
- User polls progress with `agent-factory status <name>`

**Role Decomposition Pipeline**
Multi-step LLM pipeline run by the factory boss (decomposes into tasks for factory workers):
1. **Responsibilities generation** — LLM enumerates all responsibilities the goal requires
2. **Clustering** — responsibilities grouped into roles; strongly overlapping ones combined to avoid gaps
3. **Fit check** — LLM rubric evaluates each role for: responsibility cohesion, privilege scope, name clarity. Max 2 retries with failure reason fed back into re-clustering
4. **Enrichment** (parallelizable per role) — LLM assigns: personality tone, descriptive name, tool_allowlist from researcher-determined catalog
5. **Structured output** — `RoleSpec` Pydantic model via `messages.parse()` (same pattern as BossAgent's goal decomposition). Fields: `name`, `responsibilities`, `personality_system_prompt`, `tool_allowlist`, `requires_glibc`
6. **Deterministic YAML generation** — executor writes `agents/<role>.yaml` from validated `RoleSpec` (no free-form YAML from LLM)

**Structural Role Constants**
- **Boss**: always generated, not subject to decomposition pipeline
- **Critic**: always generated with fixed skeptical/adversarial personality + goal-specific context injected into system prompt

**Personality**
- Woven into system prompt, generated by LLM (not a separate field)
- Intentional and role-appropriate character

**Tool Allowlist and Dependencies**
- Factory's **researcher agent** investigates what tools/libraries the goal requires
- Factory's **security-checker agent** audits each tool for: privilege scope, glibc dependency, known CVEs
- `requirements.txt` generated in cluster artifact with all researcher-determined packages
- Tool allowlist in generated agent YAMLs is declarative for Phase 5
- If security-checker flags a tool as requiring glibc: Dockerfile switches from slim to Ubuntu base

**Gap Detection (Post-Creation)**
- Gap-filling is user-initiated: `agent-factory add-role <cluster> <role-description>` (Phase 5 adds this CLI command)
- Mini factory pipeline: generates a new `agents/<role>.yaml`, updates `docker-compose.yml`, notifies user to restart the service

**Cluster Naming**
- `--name` flag for explicit naming; auto-slugified from goal string as fallback
- Collision policy: fail fast with clear error message unless `--force` flag is passed
- Factory home: `FACTORY_HOME` env var, default `~/.agent-factory/`
- Cluster output: `./clusters/<name>/` relative to working directory

**Dockerfile / Image Strategy**
- Base image: `python:3.12-slim` by default
- If glibc required: switches to `ubuntu:22.04` with `python3.12`
- `requirements.txt` baked at `docker compose build` time
- `launch.sh` runs `docker compose build` then `docker compose up -d` with env var validation

**CLI Feedback** — specific output formats defined in CONTEXT.md (fire-and-forget create, task-table status, cluster summary on completion)

### Claude's Discretion
- Exact task breakdown the factory boss uses to decompose a cluster-creation goal (task titles, count, assignment strategy)
- Internal structure of `factory/generator.py` utility functions
- How the executor worker materializes documents as files on disk (tool call vs direct file write)
- Fit check rubric scoring thresholds (what constitutes a failing score)
- Auto-slug truncation behavior for very long goal strings

## Summary

Phase 5 creates the factory cluster — the core product deliverable that accepts a natural-language goal and produces a fully runnable cluster artifact. The factory cluster IS a cluster: it uses the existing BossAgent, WorkerAgent, and BaseAgent heartbeat machinery without modification. The new work is: (1) a `factory/` package with the generator module and factory-specific boss subclass, (2) a factory DB seeded at `FACTORY_HOME/factory.db`, (3) the full set of artifact templates (docker-compose.yml, Dockerfile, cluster.yaml, agents/*.yaml, requirements.txt, launch.sh, db/schema.sql), and (4) the `agent-factory create|list|status|add-role` CLI commands.

The role decomposition pipeline is the technical centrepiece: a multi-step LLM sequence (responsibilities → clustering → fit check → enrichment → RoleSpec → deterministic YAML) that the factory boss orchestrates as tasks assigned to researcher, security-checker, and executor workers. All LLM structured output uses the established `messages.parse()` + Pydantic pattern already proven in BossAgent and WorkerAgent.

The factory boss is a thin BossAgent subclass: it overrides `decompose_goal` to emit factory-specific tasks (research tools, design roles, generate YAML, write docker-compose, seed DB, write launch.sh) rather than product-domain tasks. The executor worker writes artifact files directly to disk using Python `pathlib` + `shutil.copytree` for the runtime copy; this is a direct file write, not a Claude tool call.

**Primary recommendation:** Build the factory package as a thin layer on top of the existing runtime. The generator module is a collection of pure Python + Jinja2/string-template functions. The factory boss subclass is ~150 lines. Keep the architecture firmly self-similar — resist any temptation to introduce a separate non-agent codepath.

## Architecture Patterns

### Recommended Project Structure
```
factory/
├── __init__.py          # empty; marks package
├── boss.py              # FactoryBossAgent(BossAgent) — overrides decompose_goal
├── generator.py         # Pure-function artifact generators (YAML, Dockerfile, etc.)
├── models.py            # RoleSpec, RolesResult, FitCheckResult Pydantic models
└── pipeline.py          # Multi-step role decomposition pipeline (called by factory boss)
```

Note: `factory/` is a new top-level package alongside `runtime/`. The factory boss and workers reference `runtime/` types directly.

### Pattern 1: Factory Boss as BossAgent Subclass

**What:** `FactoryBossAgent` subclasses `BossAgent` and overrides only `decompose_goal`. The goal decomposition for a factory job emits a fixed set of factory-specific tasks (research, design roles, generate artifacts) rather than product-domain tasks.

**When to use:** Creating a factory cluster job — `agent-factory create` seeds the goal then spawns a FactoryBossAgent.

```python
# factory/boss.py
class FactoryBossAgent(BossAgent):
    """Factory-specific boss: decomposes cluster-creation goals into generator tasks."""

    async def decompose_goal(self, goal_id: str, goal_description: str) -> None:
        """Emit the canonical factory task set for this goal."""
        tasks = _factory_task_set(goal_description)
        for spec in tasks:
            task_id = _uuid()
            await self._insert_task(goal_id, task_id, spec, reviewer_agents=[])
```

The `_factory_task_set()` function returns a fixed list of `TaskSpec` objects tailored to artifact generation: research dependencies, design roles (runs the pipeline), generate YAML files, write docker-compose, seed DB, write launch.sh.

### Pattern 2: RoleSpec Pydantic Model via messages.parse()

**What:** Structured LLM output for each role, validated before any file is written. Follows the exact same pattern as `DecompositionResult`/`TaskSpec` in `boss.py`.

**When to use:** Step 5 of the role decomposition pipeline (enrichment → structured output).

```python
# factory/models.py
class RoleSpec(pydantic.BaseModel):
    """One agent role in the generated cluster."""
    name: str                       # slug, e.g. "pdf-extractor"
    responsibilities: list[str]
    personality_system_prompt: str  # full system prompt with embedded personality
    tool_allowlist: list[str]
    requires_glibc: bool = False

class RolesResult(pydantic.BaseModel):
    roles: list[RoleSpec]

class FitCheckResult(pydantic.BaseModel):
    passed: bool
    failing_role: str | None = None
    reason: str | None = None
```

### Pattern 3: Deterministic Artifact Generation in generator.py

**What:** Pure functions that accept structured data (RoleSpec list, cluster name, goal) and return strings or write files. No LLM calls inside generator.py — all LLM work happens in pipeline.py.

**When to use:** Executor worker calls generator functions after pipeline completes.

```python

    """Render docker-compose.yml for a cluster of given roles."""
    ...

    """Render config/cluster.yaml."""
    ...

def render_launch_sh(cluster_name: str) -> str:
    """Render launch.sh with env var validation."""
    ...

def copy_runtime(dest_dir: Path) -> None:
    """Copy runtime/ package into clusters/<name>/runtime/."""
    src = Path(__file__).parent.parent / "runtime"
```

### Pattern 4: Factory DB at FACTORY_HOME

**What:** The factory cluster uses its own SQLite database at `~/.agent-factory/factory.db` (or `$FACTORY_HOME/factory.db`). This is a regular schema-up'd cluster DB — same `DatabaseManager`, same `schema.sql`. Factory agents point their `db_path` here.

**When to use:** `agent-factory create` seeds this DB with the cluster-creation goal.

```python
# In factory CLI
factory_home = Path(os.environ.get("FACTORY_HOME", Path.home() / ".agent-factory"))
factory_db = factory_home / "factory.db"
mgr = DatabaseManager(factory_db)
await mgr.up()  # idempotent
```

### Pattern 5: fire-and-forget `agent-factory create`

**What:** Seeds factory DB with the goal, writes the cluster output directory placeholder, starts the factory heartbeat agents in a background process (or subprocess), then returns to the user immediately.

**When to use:** Every `agent-factory create` invocation.

```
agent-factory create "Build a MTG deckbuilding advisor"
→ seed factory DB
→ write clusters/mtg-deckbuilding-advisor/ (empty placeholder or scaffold)
→ start factory agents (background asyncio tasks or subprocess)
→ print:
    Factory job started: mtg-deckbuilding-advisor
      Factory DB: ~/.agent-factory/factory.db
      Track progress: agent-factory status mtg-deckbuilding-advisor
      Cluster output: ./clusters/mtg-deckbuilding-advisor/ (when complete)
→ return immediately
```

**Important implementation note:** Because `asyncio.run()` blocks until complete, the fire-and-forget model requires the factory agents to run in a detached subprocess. The recommended approach is `subprocess.Popen([sys.executable, '-m', 'factory.runner', goal_id, cluster_name])` where `factory/runner.py` is a simple async entry point that starts the boss and worker heartbeat tasks.

### Pattern 6: docker-compose.yml Generation

**What:** The generated `docker-compose.yml` has one service per role (boss, critic, plus all decomposed roles), sharing a named volume for the DB and config.

```yaml
# Generated clusters/<name>/docker-compose.yml
services:
  boss:
    build: .
    command: python -m runtime.cli run-agent --role boss
    volumes:
      - cluster-data:/data
    env_file: .env
  pdf-extractor:
    build: .
    command: python -m runtime.cli run-agent --role pdf-extractor
    volumes:
      - cluster-data:/data
    env_file: .env
  # ... one service per role

volumes:
  cluster-data:
```

**Stagger offsets:** Generated from a formula: `boss=0`, then `offset = (index * interval) / role_count` for each worker role, where interval defaults to 600s.

### Pattern 7: launch.sh with env var validation

**What:** Shell script that validates `ANTHROPIC_API_KEY` is set, then runs `docker compose build && docker compose up -d`.

```bash
#!/usr/bin/env bash
set -euo pipefail

if [[ -z "${ANTHROPIC_API_KEY:-}" ]]; then
  echo "ERROR: ANTHROPIC_API_KEY is not set. Export it before running launch.sh." >&2
  exit 1
fi

docker compose build
docker compose up -d
echo "Cluster started. Use 'cluster tasks list' to track progress."
```

### Anti-Patterns to Avoid

- **Free-form YAML from LLM:** Never let the LLM write raw YAML strings for agent configs. The LLM produces `RoleSpec` structured output; generator.py serialises it deterministically. This prevents injection attacks and malformed YAML.
- **Blocking asyncio.run() in create command:** The heartbeat loop never terminates on its own. The `agent-factory create` command must spawn agents as a subprocess (not `await agent.start()` inline) to return immediately to the user.
- **Hardcoding ANTHROPIC_API_KEY in generated files:** Generated `.env.example` must have a placeholder, never a real key. The `launch.sh` validation checks the env at runtime only.
- **Nested asyncio.run():** Consistent with Phase 1 RESEARCH decision: never nest `asyncio.run()` calls. Each CLI command wraps one coroutine.
- **Mutation of AgentConfig:** Generate new dicts for YAML, do not mutate existing AgentConfig instances.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML serialisation | Custom YAML string builder | `yaml.dump()` (pyyaml already dep) | Handles quoting, multiline strings, unicode |
| Directory copy | Manual file iteration | `shutil.copytree(src, dest, dirs_exist_ok=True)` | Handles symlinks, permissions, nested dirs |
| Slug generation | Complex regex pipeline | `re.sub(r'[^a-z0-9]+', '-', goal.lower()).strip('-')[:50]` | One line; no new dep |
| Structured LLM output | Manual JSON parsing | `messages.parse(output_format=RoleSpec)` | Already established pattern; validation built in |
| Docker Compose validation | Custom YAML schema checker | `subprocess.run(["docker", "compose", "config"])` | Official tool; used in end-to-end test |
| DB seeding | Direct SQL in CLI | `DatabaseManager.up()` + existing `_uuid()/_now_iso()` helpers | Consistent with all existing patterns |
| CLI argument parsing | argparse | `click` (already dep) | Already in use for all CLI commands |

**Key insight:** Every "new" problem in Phase 5 has an existing solution either in the project's runtime package or in the Python stdlib. The factory package is a thin orchestration layer, not a new technical domain.

## Common Pitfalls

### Pitfall 1: Blocking Create Command

**What goes wrong:** `agent-factory create` calls `asyncio.run(boss.start())` which never returns because the heartbeat loop runs indefinitely.

**Why it happens:** `BaseAgent.start()` runs `while not self._stop_event.is_set()` — it only stops on explicit signal or `CancelledError`. The CLI command must return to the shell immediately.

**How to avoid:** Use `subprocess.Popen` to launch a detached `factory/runner.py` script. The runner module is a minimal async entry point: `asyncio.run(run_factory(goal_id, cluster_name))` that starts boss + workers and awaits until the goal is complete (goal.status == 'completed'), then exits.

**Warning signs:** If `agent-factory create` hangs at the terminal, the agents are running inline rather than in a subprocess.

### Pitfall 2: Circular Import Between factory/ and runtime/

**What goes wrong:** `factory/boss.py` imports from `runtime/boss.py`, but if `runtime/__init__.py` imports from `factory/`, a circular import at collection time crashes pytest.

**Why it happens:** `runtime/` is the foundational package. `factory/` depends on `runtime/`, never the reverse.

**How to avoid:** `factory/` imports freely from `runtime/`. `runtime/` never imports from `factory/`. Keep `runtime/__init__.py` empty (as it is now).

**Warning signs:** `ImportError: cannot import name 'X' from partially initialized module`.

### Pitfall 3: LLM Writing Raw YAML for Agent Configs

**What goes wrong:** An LLM prompt asks "write the agents/researcher.yaml file" and the model produces malformed YAML, injects shell metacharacters in system prompts, or uses Python-incompatible YAML features.

**Why it happens:** LLMs are unreliable YAML serialisers for structured config files.

**How to avoid:** LLM produces only `RoleSpec` structured output (via `messages.parse()`). Generator.py calls `yaml.dump()` on validated Python dicts. Zero free-form YAML from LLM.

**Warning signs:** `yaml.scanner.ScannerError` when loading generated agent configs; system prompts containing `$(cmd)` or backtick sequences.

### Pitfall 4: Factory State File Collisions with Product Cluster State Files

**What goes wrong:** Factory agents write state files to `runtime/state/` (hardcoded in `heartbeat.py` as `STATE_DIR`). If a product cluster also runs on the same machine, state files from factory agents and product agents collide by agent_id.

**Why it happens:** `STATE_DIR = Path(__file__).parent / "state"` is global in `heartbeat.py`. Factory agents use the same runtime package.

**How to avoid:** `FACTORY_HOME` must also redirect the state directory. Pass `state_dir=factory_home / "state"` in the factory agent `AgentConfig`. The `BaseAgent` already uses `config.state_dir` if set (it uses `STATE_DIR` as module-level default but `_state_path` is computed from `config.agent_id` under `STATE_DIR`). Verify that `AgentConfig.state_dir` is propagated to `BaseAgent._state_path` correctly — this is a known concern.

**Warning signs:** State files for `factory-boss-01` appearing in the product cluster's `runtime/state/` directory.

### Pitfall 5: shutil.copytree Fails if Destination Exists

**What goes wrong:** `shutil.copytree(src, dest)` raises `FileExistsError` if `dest` already exists (Python < 3.8 behaviour) or on re-runs with `--force`.

**Why it happens:** Default `copytree` behaviour before `dirs_exist_ok` parameter was added.

**How to avoid:** Use `shutil.copytree(src, dest, dirs_exist_ok=True)`. This is available from Python 3.8+; the project requires 3.12+, so it is safe.

**Warning signs:** `FileExistsError` on `agent-factory create --force`.

### Pitfall 6: pytest Coverage Fails Due to factory/ Not in --cov Target

**What goes wrong:** `pytest --cov=runtime` only measures coverage in `runtime/`. The `factory/` package has zero coverage reported, and the 80% floor is trivially met or incorrectly measured.

**Why it happens:** `pyproject.toml` `addopts` targets `--cov=runtime` only.

**How to avoid:** Update `pyproject.toml` to `--cov=runtime --cov=factory` once the factory package exists. This is a Wave 0 task.

**Warning signs:** Coverage report shows no `factory/*.py` lines measured; overall coverage stays artificially high.

### Pitfall 7: docker-compose.yml Stagger Offsets Need to Match Role Count

**What goes wrong:** All agents start simultaneously because stagger offsets in docker-compose.yml are all zero, causing SQLite write contention on first heartbeat.

**Why it happens:** The stagger value is hardcoded or computed incorrectly when the role count changes.

**How to avoid:** Compute stagger offsets dynamically in `render_docker_compose()`: `offset_i = i * (interval_seconds / total_roles)`, where `total_roles` includes boss and critic. Cap at `interval_seconds - 1`.

**Warning signs:** Multiple agents failing with SQLite `database is locked` errors within the first minute.

### Pitfall 8: pytest.importorskip Must Be Inside Test Body for factory/ Stubs

**What goes wrong:** If tests import `factory.generator` at module level before the package exists, pytest collection crashes for all tests in the file.

**Why it happens:** Phase 5 follows the same TDD/Wave pattern as previous phases — factory/ starts absent; tests are written as RED stubs first.

**How to avoid:** Use `pytest.importorskip` inside test body or use `try/except ImportError` in conftest (same pattern as `_has_runtime` guard). Do NOT import factory modules at test module level until implementation is in place.

**Warning signs:** `ModuleNotFoundError: No module named 'factory'` crashes the entire test session during collection.

```python
import asyncio, sys
from factory.boss import FactoryBossAgent
from runtime.config import AgentConfig

async def run_factory(goal_id: str, db_path: str) -> None:
    config = AgentConfig(agent_id="factory-boss", agent_role="boss", db_path=db_path)
```

4. Print: "Role added. Restart the cluster with `./launch.sh` to activate."

## Open Questions

1. **Factory boss termination: how does the factory boss know the cluster is complete?**
   - What we know: `_gap_fill_and_completion_check` marks `goal.status = 'completed'` when LLM judges goal achieved; `BaseAgent.start()` loops on `_stop_event`
   - What's unclear: Factory runner.py needs to detect goal completion and stop the heartbeat loop without modifying BaseAgent
   - Recommendation: `FactoryBossAgent` overrides `do_own_tasks` to call `_gap_fill_and_completion_check` and then sets `self._stop_event.set()` when goal is completed. OR: factory/runner.py polls goal status every N seconds and sends SIGTERM to the process.

2. **factory/runner.py subprocess: Windows compatibility of Popen with detachment**
   - What we know: Windows does not support `os.fork()`; `subprocess.Popen` works on Windows via `CREATE_NEW_PROCESS_GROUP` or `DETACHED_PROCESS`
   - What's unclear: The project is developed on Windows (platform: win32 per env context) but targets Docker containers (Linux). The factory agents run inside Docker for product use, but during development `agent-factory create` runs on Windows
   - Recommendation: Use `subprocess.Popen([sys.executable, '-m', 'factory.runner', ...], close_fds=True)` without platform-specific flags — let the test suite on Windows validate. If daemonisation is needed, investigate `pythonw.exe` or restructure to use `asyncio.create_subprocess_exec`.

3. **Runtime copy: should copied runtime/ be a pip-installed package or a raw directory copy?**
   - What we know: CONTEXT.md specifies `shutil.copytree` copy of `runtime/` into cluster artifact; the Dockerfile does `COPY runtime/ ./runtime/`
   - What's unclear: If the generated cluster needs `python -m runtime.cli`, the `runtime/` directory must be on the Python path. Putting it in `/app/` and running `python -m runtime.cli` from `/app/` should work
   - Recommendation: Copy to `/app/runtime/`, WORKDIR `/app/`, CMD runs `python -m runtime.cli`. No pip install of the runtime needed inside the container.
